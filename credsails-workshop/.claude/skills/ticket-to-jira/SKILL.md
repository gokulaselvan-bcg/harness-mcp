# Ticket to Jira Export Workflow

This workflow takes ticket files generated by the `/blueprint-to-tickets` command and pushes them to Jira using MCP tools. This is part of the **Lead Workflow**.

## Skills to Invoke

**IMPORTANT: You MUST first read and invoke these skills before proceeding:**

1. `managing-jira` - For all Jira MCP operations (create issues, transitions, etc.)

## Atlassian MCP Prerequisites

**CRITICAL: This workflow requires the Atlassian MCP (Model Context Protocol) server to interact with Jira.**

### Step 0: Verify Atlassian MCP Availability

**Before proceeding with this workflow, verify that Atlassian MCP tools are available:**

1. **Check if Atlassian MCP tools are accessible:**
   - Attempt to call `mcp_atlassian_atlassianUserInfo()` (no parameters required)
   - This tool should return user information if MCP is properly configured and authenticated

2. **If MCP tools are NOT available or authentication fails:**
   - Display error: "⚠️ Atlassian MCP is not configured or not authenticated"
   - **Stop the workflow immediately** - do not attempt alternative methods
   - Provide user guidance:
     ```
     The Atlassian MCP server is required for this workflow but is not currently available.

     To configure Atlassian MCP:
     1. Ensure the Atlassian MCP server is configured in your Cursor settings
     2. Authenticate with your Atlassian account
     3. Verify access by checking MCP server status in Cursor settings

     Once configured and authenticated, restart this workflow.
     ```
   - **Exit workflow** - do not proceed

3. **If MCP tools are available:**
   - Display confirmation: "✓ Atlassian MCP is configured and authenticated"
   - Store user info for reference (optional)
   - Proceed to Step 1

**Important:**
- This check happens BEFORE any other workflow steps
- If MCP is not available, the workflow MUST NOT proceed
- Do not attempt to use command-line tools, gh CLI, or other alternative methods for Jira access
- The workflow explicitly requires MCP - other methods are not supported

## Step 1: Identify and Validate Ticket File Structure

**Actions:**
1. List all `*-tickets.md` files under `artifacts/blueprints/**/tickets/`
2. Present the list to the user with clear identifiers
3. Ask user to select which tickets file to export

**If no ticket files found:**
- Inform the user that no ticket files were found
- **DO NOT PROCEED** - Exit the workflow
- Recommend: "Please use the `/blueprint-to-tickets` command to create tickets from a blueprint first. Once tickets are created, you can use this workflow to export them to Jira."
- Do not ask for alternative paths or proceed with any other steps

**User selection format:**
- Accept filename (e.g., `694159c2-e6ef-1643-8208-8f1482fe38f0-tickets.md`)
- Accept full path if needed

**Expected Ticket File Structure:**
The tickets follow a specific structure created by the `/blueprint-to-tickets` command:

1. **Top-level tickets file** (e.g., `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets.md`):
   - Contains YAML frontmatter with metadata:
     ```yaml
     ---
     generated: 2025-01-20
     source_blueprint: blueprint-[PROJECT-ID]-[description].md
     blueprint_id: [PROJECT-ID]
     total_tickets: 85
     epics_count: 11
     ---
     ```
   - Contains "Epic Tickets" section with all epic definitions
   - Each epic has format: `- [ ] **EPIC-[NUMBER]-[SHORT-NAME]: [Epic Name]**`
   - Epics include: Issue Type, Summary, Effort Estimate, Labels, Description, Scope, Key Deliverables, Dependencies, Acceptance Criteria, Related Blueprint Sections, Notes

2. **Subdirectory** (e.g., `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets/`):
   - Contains individual epic files: `EPIC-[NUMBER]-[SHORT-NAME].md`
   - Each epic file contains all tickets for that epic
   - Tickets have format: `- [ ] **[TICKET-ID]: [Ticket Title]**`
   - Tickets include: Issue Type, Summary, Effort Estimate, Labels, Parent Epic, Description, Acceptance Criteria, Technical Details, Dependencies, Testing Requirements, Related Sections, Implementation Notes, Definition of Done

3. **Validation:**
   - Verify the selected file matches the expected structure
   - Check that the subdirectory exists and contains epic files
   - Report any structural issues before proceeding

## Step 2: Read and Parse Top-Level Tickets File

**Actions:**
1. Read the selected top-level tickets file (e.g., `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets.md`)
2. Extract metadata from frontmatter:
   - `generated`, `source_blueprint`, `blueprint_id`, `total_tickets`, `epics_count`
3. Parse the "Epic Tickets" section to extract all epics:
   - Look for epic sections starting with `- [ ] **EPIC-[NUMBER]-[SHORT-NAME]: [Epic Name]**`
   - Extract epic metadata:
     - Epic ID (e.g., `EPIC-01-DEV-SETUP`)
     - Epic Name
     - Issue Type (should be "Epic")
     - Summary
     - Effort Estimate
     - Labels (comma-separated)
     - Description
     - Scope (list items)
     - Key Deliverables
     - Dependencies
     - Acceptance Criteria
     - Related Blueprint Sections
     - Notes

**Epic parsing pattern:**
- Epic sections start with `- [ ] **EPIC-[NUMBER]-[SHORT-NAME]: [Epic Name]**`
- Followed by fields in format `**Field Name:** value`
- Fields are separated by blank lines
- Epic sections end with `---` separator

## Step 3: Validate Epic Structure and Files

**Actions:**
1. Verify all epics have required fields:
   - Epic ID (EPIC-XX-XXX format)
   - Summary
   - Description
2. Check that corresponding epic files exist in subdirectory:
   - For epic `EPIC-01-DEV-SETUP`, check for `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets/EPIC-01-DEV-SETUP.md` (i.e., in the `{id}-tickets` subdirectory next to the selected top-level tickets file)
   - List all epic files found
   - Report any missing epic files or malformed epics
3. Display structure summary:
   - Total epics found: `[N]`
   - Epic files found: `[list of files]`
   - Missing epic files: `[list of missing files]` (if any)
4. Ask user to confirm structure is correct before proceeding

## Step 4: Get Jira Project/Board Selection and Cloud ID

**CRITICAL: This step MUST be completed before any tickets are created. The user must explicitly select a Jira project/board.**

**Actions:**
1. **Get Cloud ID:**
   - Use MCP tool `mcp_atlassian_getAccessibleAtlassianResources()` (no parameters)
   - Response contains array of resources with `id` field (cloud ID)
   - Extract first resource's `id` as cloud ID
   - Store cloud ID for subsequent API calls

2. **Get Available Projects:**
   - Use MCP tool `mcp_atlassian_getVisibleJiraProjects(cloudId=cloud_id, action="create")`
   - Parameters:
     - `cloudId`: Cloud ID from step 1
     - `action`: "create" (to get projects where user can create issues)
   - Response contains array of projects with `key` and `name` fields

3. **Present Project List to User:**
   - Display all available projects with clear formatting:
     ```
     Available Jira Projects:
     1. [PROJECT-KEY] - [PROJECT-NAME]
     2. [PROJECT-KEY] - [PROJECT-NAME]
     ...
     ```
   - Include project key and name for each project
   - Show total count of available projects

4. **Ask User to Select Project:**
   - **Prompt user explicitly**: "Which Jira project do you want to add tickets to? Please enter the project KEY (e.g., '[DEFAULT_JIRA_PROJECT_KEY]' from AGENTS.md) or number from the list above."
   - Accept project key or number from the list
   - If user enters a number, match it to the project from the list
   - If user enters a project key, validate it exists in the available projects
   - **DO NOT PROCEED** until user explicitly selects a project

5. **Validate Selected Project:**
   - Get issue types for the selected project using `mcp_atlassian_getJiraProjectIssueTypesMetadata(cloudId=cloud_id, projectIdOrKey=selected_project_key)`
   - Verify that "Epic" and "Task" issue types are available
   - Check that issue type names match exactly ("Epic" and "Task")
   - If project doesn't have required issue types, report error:
     ```
     Error: Project [PROJECT-KEY] does not have required issue types (Epic, Task).
     Please select a different project.
     ```
     - Present the project list again and ask user to select a different project
     - **DO NOT PROCEED** until user selects a valid project

6. **Confirm Selection:**
   - Display confirmation summary:
     ```
     Selected Jira Project:
     ├─ Project Key: [PROJECT-KEY]
     ├─ Project Name: [PROJECT-NAME]
     ├─ Cloud ID: [CLOUD-ID]
     └─ Available Issue Types: Epic, Task ✓
     ```
   - Ask user to confirm: "Is this the correct project? (yes/no)"
   - If user says "no", return to step 3 (present project list again)
   - If user says "yes", proceed to Step 5
   - **DO NOT PROCEED** to ticket creation until user confirms

**Default project handling:**
- If a default project is configured (use the Default Jira Project Key from AGENTS.md), suggest it but still require user confirmation
- Do not auto-select any project - always require explicit user selection

**MCP Tool Usage:**
```python
# Step 1: Get cloud ID

cloud_id = resources[0]["id"]

# Step 3: Get projects
projects = mcp_atlassian_getVisibleJiraProjects(cloudId=cloud_id, action="create")

# Step 4: Validate issue types
issue_types = mcp_atlassian_getJiraProjectIssueTypesMetadata(
    cloudId=cloud_id,
    projectIdOrKey="[DEFAULT_JIRA_PROJECT_KEY]"  # Get from AGENTS.md
)
# Check that "Epic" and "Task" are in the issue types list
```

**Important Notes:**
- Always ask the user to select a project explicitly - never auto-select
- Project selection must happen BEFORE any ticket creation
- If user wants to change project midway, they must restart the workflow
- Validate project has required issue types (Epic, Task) before proceeding

## Step 5: Main Epic Processing Loop

**High-Level Loop Structure (Enumerated):**

The main loop processes epics sequentially. For each epic in the top-level file:

1. **Create Epic in Jira**
   - Extract epic metadata from top-level file
   - Create epic using MCP `mcp_atlassian_createJiraIssue`
   - Store epic issue key for linking tickets

2. **Check Epic Checkbox in Top-Level File**
   - Update `- [ ]` to `- [x]` for the epic entry
   - Save the updated top-level file

3. **Read Epic's Ticket File**
   - Read corresponding file from subdirectory: `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets/EPIC-[NUMBER]-[SHORT-NAME].md`
   - Parse all tickets in the file

4. **Create All Tickets for Epic (Nested Loop)**
   - For each ticket in the epic file:
     a. Create ticket in Jira using MCP `mcp_atlassian_createJiraIssue` (without parent link)
     b. Link ticket to parent epic using MCP `mcp_atlassian_editJiraIssue` with `parent` field (using epic issue key)
     c. Check ticket checkbox in epic file (`- [ ]` to `- [x]`)
     d. Save the updated epic file
     e. Report success/failure

5. **Report Epic Progress**
   - Display summary of created tickets
   - Ask user to continue to next epic or stop

6. **Move to Next Epic**
   - Repeat steps 1-5 for next epic
   - Continue until all epics are processed

**Detailed Steps for Each Epic:**

### Loop: For Each Epic in Top-Level File

#### 5.1: Create Epic in Jira

**Actions:**
1. Display current epic being processed: `Processing Epic: [EPIC-ID] - [Epic Name]`
2. Extract epic fields from top-level file:
   - `summary`: Epic Name (max 255 characters)
   - `description`: Epic Description + Scope + Key Deliverables + Acceptance Criteria (formatted as Markdown)
   - `labels`: Parse comma-separated labels into array (e.g., `["infrastructure", "setup", "development"]`)
   - `issuetype`: "Epic"
3. Use MCP tool `mcp_atlassian_createJiraIssue` to create epic:
   - `cloudId`: Cloud ID from Step 4
   - `projectKey`: Selected project key from Step 4
   - `issueTypeName`: "Epic" (must match exactly)
   - `summary`: Epic Name (required, max 255 characters)
   - `description`: Formatted epic description (Markdown, optional)
   - `additional_fields`: Object with labels array if labels exist: `{"labels": ["label1", "label2"]}`
4. Extract issue key from response:
   - Response contains `key` field (e.g., `"[DEFAULT_JIRA_PROJECT_KEY]-123"`)
   - Store created epic's issue key for linking tickets
5. **Check checkbox in top-level file:**
   - Update the epic entry in the top-level file: Change `- [ ]` to `- [x]` for the epic that was just created
   - Save the updated top-level file
6. Report success/failure to user: `✓ Epic created: [EPIC-KEY] - [Epic Name]` or `✗ Failed to create epic: [error]`

**MCP Tool Usage:**
```python
# Create epic
epic_response = mcp_atlassian_createJiraIssue(
    cloudId=cloud_id,
    projectKey="[DEFAULT_JIRA_PROJECT_KEY]",  # Get from AGENTS.md
    issueTypeName="Epic",
    summary="Development Environment Setup",
    description="## Description\n\nThis epic establishes...",
    additional_fields={"labels": ["infrastructure", "setup"]}
)
epic_key = epic_response["key"]  # e.g., "[DEFAULT_JIRA_PROJECT_KEY]-123"
```

**Epic description format:**
```markdown
## Description
[Epic description text]

## Scope
- [Item 1]
- [Item 2]

## Key Deliverables
- [Deliverable 1]
- [Deliverable 2]

## Dependencies
- Depends on: [Other epics]
- Blocks: [Other epics]

## Acceptance Criteria
- [ ] Criterion 1
- [ ] Criterion 2

## Related Blueprint Sections
[Section references]

## Notes
[Notes text]
```

#### 5.2: Read Epic Ticket File

**Actions:**
1. Read the corresponding epic file from subdirectory:
   - `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets/EPIC-[NUMBER]-[SHORT-NAME].md`
2. Parse all tickets in the file:
   - Look for ticket sections starting with `- [ ] **[TICKET-ID]: [Ticket Title]**`
   - Extract ticket metadata for each ticket:
     - Ticket ID (e.g., `DEV-SETUP-001`)
     - Ticket Title
     - Issue Type (Task or Subtask)
     - Summary
     - Effort Estimate
     - Labels
     - Parent Epic (should match current epic)
     - Description
     - Acceptance Criteria
     - Technical Details
     - Dependencies
     - Testing Requirements
     - Related Sections
     - Implementation Notes
     - Definition of Done
3. Display ticket count: `Found [N] tickets in epic file`

**Ticket parsing pattern:**
- Ticket sections start with `- [ ] **[TICKET-ID]: [Ticket Title]**`
- Followed by fields in format `**Field Name:** value`
- Fields are separated by blank lines
- Ticket sections end with `---` separator

#### 5.3: Create Tickets in Jira for Epic (Nested Loop)

**For each ticket in the epic file (process in order):**

**Actions:**
1. Display current ticket: `  Creating ticket: [TICKET-ID] - [Ticket Title]`
2. Extract ticket fields:
   - `summary`: Ticket Title (max 255 characters)
   - `description`: Full ticket description (formatted as Markdown)
   - `labels`: Parse comma-separated labels into array (e.g., `["backend", "setup"]`)
   - `issuetype`: "Task"
   - Epic issue key from Step 5.1 (e.g., `"[DEFAULT_JIRA_PROJECT_KEY]-123"`) - stored for linking after creation
3. Use MCP tool `mcp_atlassian_createJiraIssue` to create ticket (without parent link):
   - `cloudId`: Cloud ID from Step 4
   - `projectKey`: Selected project key from Step 4
   - `issueTypeName`: "Task" (must match exactly)
   - `summary`: Ticket Title (required, max 255 characters)
   - `description`: Formatted ticket description (Markdown, optional)
   - `additional_fields`: Object with labels array if labels exist: `{"labels": ["label1", "label2"]}`
   - **Note:** Do NOT include `parent` field in `createJiraIssue` - linking happens after creation
4. Extract issue key from response:
   - Response contains `key` field (e.g., `"[DEFAULT_JIRA_PROJECT_KEY]-124"`)
   - Store created ticket's issue key for future reference
5. **Link ticket to parent epic using `mcp_atlassian_editJiraIssue`:**
   - Use `mcp_atlassian_editJiraIssue` to update the ticket with parent epic link
   - `cloudId`: Cloud ID from Step 4
   - `issueIdOrKey`: Created ticket's issue key (e.g., `"[DEFAULT_JIRA_PROJECT_KEY]-124"`)
   - `fields`: Object with `parent` field: `{"parent": {"key": "[DEFAULT_JIRA_PROJECT_KEY]-123"}}`
   - This links the ticket to the epic as a child
6. **Check checkbox in epic file:**
   - Update the ticket entry in the epic file: Change `- [ ]` to `- [x]` for the ticket that was just created
   - Save the updated epic file
7. Report success/failure: `    ✓ Ticket created: [TICKET-KEY]` or `    ✗ Failed to create ticket: [error]`
8. Continue to next ticket in the epic file

**MCP Tool Usage:**
```python
# Step 1: Create task ticket (without parent link)
ticket_response = mcp_atlassian_createJiraIssue(
    cloudId=cloud_id,
    projectKey="[DEFAULT_JIRA_PROJECT_KEY]",  # Get from AGENTS.md
    issueTypeName="Task",
    summary="Backend Development Environment Setup and Verification",
    description="## Description\n\nVerify and complete...",
    additional_fields={"labels": ["backend", "setup"]}
    # Note: Do NOT include parent field here - linking happens after creation
)

ticket_key = ticket_response["key"]  # e.g., "[DEFAULT_JIRA_PROJECT_KEY]-124"

# Step 2: Link ticket to parent epic using editJiraIssue
mcp_atlassian_editJiraIssue(
    cloudId=cloud_id,
    issueIdOrKey=ticket_key,  # e.g., "[DEFAULT_JIRA_PROJECT_KEY]-124"
    fields={"parent": {"key": "[DEFAULT_JIRA_PROJECT_KEY]-123"}}  # Epic issue key
)
```

**Important Notes:**
- **Linking Tasks to Epics**: Tasks are linked to Epics using a two-step process:
  1. First, create the ticket using `mcp_atlassian_createJiraIssue` (without parent field)
  2. Then, link it to the epic using `mcp_atlassian_editJiraIssue` with the `parent` field set to the epic's key
- This approach works for next-gen Jira projects where tasks can have epics as parents
- Issue type names must match exactly: "Epic" and "Task" (case-sensitive)
- Labels are passed as an array in `additional_fields` object: `{"labels": ["label1", "label2"]}`
- If Epic linking fails, the ticket will be created but not linked to the epic - log this as a warning and continue

**Handle ticket dependencies:**
- If ticket references other tickets (e.g., "Blocks: DEV-SETUP-002"), store the reference
- After all tickets are created, optionally create links between tickets using Jira's link functionality
- For now, keep dependencies in description text (can enhance later with actual Jira links)

#### 5.4: Report Epic Progress

**After creating all tickets for an epic:**
1. Display epic completion summary:
   ```
   Epic Complete: [EPIC-KEY] - [Epic Name]
   ├─ Epic created: [EPIC-KEY]
   ├─ Tickets created: [N] tickets
   ```

2. Count successful vs failed tickets
3. Ask user if they want to continue to next epic or stop:
   - If user says "continue" or "next": Proceed to Step 5.5

#### 5.5: Move to Next Epic

**Actions:**
1. Move to next epic in the list from top-level file
2. Repeat Steps 5.1 through 5.4 for the next epic
3. Continue until all epics are processed
4. If all epics are complete, proceed to Step 6 (Final Summary)

## Step 6: Final Summary

**After all epics and tickets are created:**

**Actions:**
1. Generate final summary:
   - Total epics created: `[N]`
   - Total tickets created: `[M]`
   - Jira project: `[PROJECT-KEY]`
   - Epic issue keys: List all epic keys
2. Create a mapping file:
   - `artifacts/blueprints/{id}--{name}/tickets/{id}-tickets/jira-mapping.json`
   - Maps ticket IDs to Jira issue keys:
     ```json
     {
       "epics": {
         "EPIC-01-DEV-SETUP": "[DEFAULT_JIRA_PROJECT_KEY]-123",
         "EPIC-02-BLOB-STORAGE": "[DEFAULT_JIRA_PROJECT_KEY]-124"

         "DEV-SETUP-002": "[DEFAULT_JIRA_PROJECT_KEY]-126"
       }
     }
     ```
3. Display summary to user
4. Ask if user wants to view tickets in Jira

## Error Handling

**During epic creation:**
- If epic creation fails, report error and ask user if they want to continue with next epic
- Store failed epics for retry

**During ticket creation:**
- If ticket creation fails, report error and continue with next ticket
- Store failed tickets for retry
- Don't fail entire epic if one ticket fails

**Common errors to handle:**
- Invalid project key
- Missing required fields
- Epic/ticket limit exceeded
- Network/API errors
- Authentication errors

## Best Practices

1. **Batch Processing**: Process epics one at a time, but create all tickets for an epic before moving to next
2. **Error Recovery**: Continue processing even if individual tickets fail
3. **Progress Reporting**: Report progress after each epic completion
4. **Mapping File**: Save Jira issue key mappings for future reference
5. **Description Formatting**: Preserve Markdown formatting in Jira descriptions
6. **Label Handling**: Parse and apply labels correctly (array format)
7. **Parent Linking**: Ensure all tickets are linked to their parent epic

## Notes

- This workflow does NOT execute MCP tools by default - it validates the workflow first
- User must explicitly confirm before any Jira API calls are made
- All parsing and validation happens before any MCP tool calls
- **Checkbox checking**: As epics and tickets are created in Jira, the corresponding checkboxes in the ticket files are automatically updated:
  - Epic checkboxes are updated in the top-level file (`artifacts/blueprints/{id}--{name}/tickets/{id}-tickets.md`)
  - Ticket checkboxes are updated in their respective epic files (`artifacts/blueprints/{id}--{name}/tickets/{id}-tickets/EPIC-XX-XXX.md`)
  - This provides visual tracking of progress in the markdown files
- Mapping file is created for tracking and potential future updates
- Files are saved after each epic and ticket creation to ensure progress is preserved
